# رصدلینک — بک‌اند

بک‌اندی برای رصد خودکار تغییرات محتوایی سایت‌ها: هر لینکی که ثبت کنید هر ۱۲ دقیقه
یک‌بار چک می‌شود، اگر **محتوای واقعی** به صفحه اضافه شده باشد (نه تبلیغات، نه
شمارنده‌های تعاملی مثل لایک/نظر/بازدید) یک اسکرین‌شات کامل از صفحه گرفته و در
دیتابیس ذخیره می‌شود؛ اسکرین‌شات قبلیِ همان لینک هم‌زمان حذف می‌شود (طبق خواسته‌ی
پروژه فقط آخرین عکس نگه داشته می‌شود).

## معماری

```
Node.js (Express)  →  PostgreSQL
        │
        ├── node-cron هر ۱۲ دقیقه اجرا می‌شود
        ├── Puppeteer صفحه را رندر می‌کند (برای سایت‌های جاوااسکریپتی هم کار می‌کند)
        ├── @mozilla/readability + jsdom محتوای اصلی را استخراج می‌کنند
        │     (نویگیشن، سایدبار، تبلیغات، نظرات به‌طور خودکار حذف می‌شوند)
        ├── فیلتر سفارشی (contentFilter.js) باقیمانده‌ی نویز تبلیغاتی/تعاملی را پاک می‌کند
        ├── diffService.js فقط پاراگراف‌های *جدید* را با نسخه‌ی قبلی مقایسه می‌کند
        └── در صورت تغییر معنادار → اسکرین‌شات + حذف عکس قبلی + رکورد در دیتابیس + اعلان
```

## چرا این‌طور تشخیص می‌دهد که تغییر «واقعی» است؟

1. **Readability** با الگوریتم همان چیزی که فایرفاکس در Reader Mode استفاده می‌کند،
   محتوای اصلی مقاله/صفحه را از نویگیشن، سایدبار، فوتر و اکثر تبلیغات جدا می‌کند.
2. پیش از آن هم یک لیست از سلکتورهای پرتکرار تبلیغاتی/نظرات/اشتراک‌گذاری حذف می‌شوند
   (`src/services/scraperService.js` → `JUNK_SELECTORS`).
3. متن باقی‌مانده به پاراگراف تقسیم و با نسخه‌ی ذخیره‌شده‌ی قبلی **فقط برای یافتن
   پاراگراف‌های اضافه‌شده** مقایسه می‌شود (نه جابه‌جایی/حذف).
4. پاراگراف‌های اضافه‌شده یک بار دیگر با الگوهای فارسی/انگلیسیِ تبلیغ و تعامل
   فیلتر می‌شوند (`src/services/contentFilter.js` — قابل ویرایش و گسترش).
5. اگر حجم متن باقی‌مانده از آستانه‌ی `MIN_ADDED_CHARS` (پیش‌فرض ۴۰ کاراکتر) بیشتر
   بود، به‌عنوان «تغییر محتوایی واقعی» ثبت می‌شود.

> این روش قانون‌محور (rule-based) و بدون هزینه‌ی API خارجی است. اگر بعداً خواستید
> دقت را با هوش مصنوعی (مثلاً تشخیص هوشمندتر تبلیغ/تعامل با Claude API) بالاتر
> ببرید، دقیقاً همین‌جا (`diffService.js`) نقطه‌ی اتصال است.

## پیش‌نیازها

- Node.js نسخه ۱۸ به بالا
- PostgreSQL (نسخه ۱۳ به بالا کافی است)
- برای اجرای Puppeteer روی لینوکس ممکن است نیاز به نصب چند پکیج سیستمی سطح پایین
  (کتابخانه‌های کروم headless) باشد؛ در Docker/Ubuntu معمولاً با
  `npx puppeteer browsers install chrome` یا نصب `chromium` سیستمی حل می‌شود.

## نصب و اجرا

```bash
cd rasadlink-backend
npm install

cp .env.example .env
# سپس .env را باز کنید و DATABASE_URL و بقیه مقادیر را مطابق محیط خودتان تنظیم کنید

# ساخت دیتابیس (اگر از قبل نساخته‌اید)
createdb rasadlink

# اجرای مایگریشن (ساخت جدول‌ها)
npm run migrate

# اجرای سرور (شامل زمان‌بند خودکار)
npm start
# یا برای توسعه با ری‌استارت خودکار:
npm run dev
```

سرور روی `http://localhost:4000` بالا می‌آید و از همان لحظه، زمان‌بند cron با
الگوی `CHECK_CRON` (پیش‌فرض هر ۱۲ دقیقه) فعال می‌شود.

برای تست فوری بدون صبر کردن تا نوبت بعدی:

```bash
npm run check-now
```

## فرانت‌اند

فایل `frontend/index.html` همان فایلی است که فرستادید، با این تفاوت که به‌جای
آرایه‌ی mock، مستقیماً از این بک‌اند داده می‌گیرد. کافی‌ست آن را با یک سرور استاتیک
ساده باز کنید (یا مستقیم در مرورگر) و مطمئن شوید بک‌اند روی `http://localhost:4000`
در حال اجراست. اگر بک‌اند را روی آدرس دیگری اجرا می‌کنید، قبل از تگ اسکریپت اصلی
این خط را اضافه کنید:

```html
<script>window.RASADLINK_API_BASE = 'https://api.example.com';</script>
```

تغییرات اعمال‌شده روی فرانت:
- گرفتن لیست لینک‌ها، افزودن لینک جدید، و باز کردن جزئیات هر کارت (اسکرین‌شات +
  تاریخچه) همگی به API واقعی وصل شده‌اند.
- کلیک روی هر کارت، بج «تغییر جدید» را خاموش می‌کند (اعلان‌های آن لینک را
  خوانده‌شده علامت می‌زند).
- دو آیتم منوی کناری «نوتیفیکیشن‌ها» و «تاریخچه تغییرات» یک پنل ساده باز می‌کنند
  که از API واقعی پر می‌شود.
- بخش «تنظیمات حساب» و «خروج» چون در پروژه سیستم کاربری/لاگین وجود نداشت، دست‌نخورده
  باقی مانده‌اند (این پروژه فعلاً تک‌کاربره فرض شده — بخش بعدی را ببینید).

## نکات مهم و فرضیات

- **تک‌کاربره:** چون در فرانت اثری از صفحه‌ی ورود/ثبت‌نام نبود، این بک‌اند فرض
  می‌کند یک کاربر از آن استفاده می‌کند (بدون احراز هویت). اگر بعداً چند کاربر
  لازم شد، افزودن جدول `users` و یک میدل‌ور JWT ساده به این ساختار راحت است.
- **حذف عکس قبلی:** طبق خواسته‌ی شما، برای هر لینک فقط **یک** اسکرین‌شات روی دیسک
  نگه داشته می‌شود؛ با هر تغییر محتوایی جدید، فایل قبلی بلافاصله حذف و فایل جدید
  جایگزین می‌شود (`screenshotService.js`). تاریخچه‌ی متنیِ تغییرات (بدون تصویر)
  در جدول `change_logs` باقی می‌ماند تا بتوانید روند تغییرات را ببینید.
- **مقیاس‌پذیری:** بررسی لینک‌ها با `p-limit` هم‌زمان محدود می‌شود (پیش‌فرض ۳ تا
  هم‌زمان) تا فشار زیادی روی CPU/RAM سرور (به‌خصوص از طرف Puppeteer) نیاید. با
  زیاد شدن تعداد لینک‌ها، `CHECK_CONCURRENCY` را در `.env` تنظیم کنید.
- **سایت‌های محافظت‌شده در برابر ربات:** اگر سایتی درخواست‌های Puppeteer را بلاک
  کند (Cloudflare و مشابه)، آن بررسی با خطا لاگ می‌شود ولی کل چرخه متوقف نمی‌شود.

## راه‌اندازی رایگان (سرور، هاست، دیتابیس)

این بخش یک مسیر کاملاً رایگان برای بالا آوردن پروژه توضیح می‌دهد. سه سرویس رایگان
لازم دارید (هر سه بدون نیاز به کارت اعتباری برای شروع):

| نیاز | سرویس پیشنهادی | چرا |
|---|---|---|
| اجرای بک‌اند (Node.js) | **Render** (Free Web Service) | دیپلوی مستقیم از گیت‌هاب، ۷۵۰ ساعت رایگان در ماه |
| دیتابیس PostgreSQL + فضای ذخیره‌ی عکس | **Supabase** (Free) | هم Postgres رایگان می‌دهد هم Storage رایگان برای عکس‌ها، در یک حساب |
| اجرای دوره‌ای هر ۱۲ دقیقه | **cron-job.org** (رایگان) | چون سرویس رایگان Render بعد از ۱۵ دقیقه بی‌فعالیتی می‌خوابد؛ با فاصله‌ی ۱۲ دقیقه‌ای، سرور اصلاً فرصت خوابیدن پیدا نمی‌کند |
| فرانت‌اند (فایل استاتیک) | **Render Static Site** یا Netlify/Cloudflare Pages | رایگان و همیشه بیدار (فایل استاتیک نمی‌خوابد) |

> ⚠️ نکته‌ی مهم درباره‌ی محدودیت‌های رایگان: پایگاه‌داده‌ی رایگان Supabase بعد از
> **یک هفته بدون هیچ فعالیتی** موقتاً غیرفعال (pause) می‌شود — با اولین کوئری دوباره
> فعال می‌شود، فقط چند ثانیه طول می‌کشد. سرویس رایگان Render هم بعد از ۱۵ دقیقه
> بی‌فعالیتی می‌خوابد و بیدار شدنش ۳۰ تا ۶۰ ثانیه طول می‌کشد. برای یک ابزار
> شخصی/رصد پس‌زمینه (نه یک سرویس کاربر-محور با ترافیک زیاد) این محدودیت‌ها معمولاً
> بی‌اهمیت‌اند.

### مرحله ۱ — دیتابیس و فضای ذخیره‌ی عکس (Supabase)

1. در [supabase.com](https://supabase.com) ثبت‌نام کنید و یک پروژه‌ی جدید (رایگان) بسازید.
2. از منوی **Project Settings → Database**، مقدار **Connection string** (نوع URI،
   حالت *Session pooler* یا *Transaction pooler*) را کپی کنید — این می‌شود
   `DATABASE_URL`.
3. از منوی **Storage**، یک باکت (bucket) جدید به نام `screenshots` بسازید و آن را
   **Public** کنید (تا فرانت بتواند عکس‌ها را مستقیم نمایش دهد).
4. از منوی **Project Settings → API**، مقدار **Project URL** (می‌شود `SUPABASE_URL`)
   و **service_role key** (می‌شود `SUPABASE_SERVICE_ROLE_KEY` — این کلید مخفی است،
   جایی غیر از تنظیمات سرور قرار ندهید) را کپی کنید.

### مرحله ۲ — بک‌اند (Render)

1. کد را در یک ریپوی GitHub (خصوصی یا عمومی) push کنید.
2. در [render.com](https://render.com) با گیت‌هاب وارد شوید → **New → Web Service** →
   ریپوی خودتان را انتخاب کنید.
3. تنظیمات:
   - **Root Directory**: اگر بک‌اند در پوشه‌ی `rasadlink-backend` است، همین را وارد کنید.
   - **Build Command**: `npm install`
   - **Start Command**: `npm start`
   - **Instance Type**: Free
4. در بخش **Environment**، متغیرهای زیر را اضافه کنید (مقادیر را از `.env.example`
   الگو بگیرید):

   ```
   DATABASE_URL=<از Supabase>
   PUBLIC_BASE_URL=<بعد از اولین دیپلوی، آدرس Render را اینجا بگذارید>
   CORS_ORIGIN=<آدرس فرانت‌اند شما>
   STORAGE_DRIVER=supabase
   SUPABASE_URL=<از Supabase>
   SUPABASE_SERVICE_ROLE_KEY=<از Supabase>
   SUPABASE_BUCKET=screenshots
   ENABLE_INTERNAL_SCHEDULER=false
   CRON_SECRET=<یک رشته‌ی تصادفی طولانی بسازید، مثلاً با openssl rand -hex 32>
   CHECK_CONCURRENCY=1
   ```

   `ENABLE_INTERNAL_SCHEDULER=false` را حتماً بگذارید — چون روی هاست رایگانی که
   می‌خوابد، زمان‌بند داخلی قابل‌اتکا نیست؛ در مرحله‌ی بعد جایگزینش می‌کنیم.
   `CHECK_CONCURRENCY=1` هم برای اینکه Puppeteer با RAM محدود سرویس رایگان
   (معمولاً ۵۱۲ مگابایت) اذیت نکند.

5. Deploy کنید. نیازی به اجرای دستی مایگریشن نیست — چون **Shell روی پلن رایگان
   Render در دسترس نیست**، دستور `npm start` طوری تنظیم شده که هر بار سرور بالا
   می‌آید، اول مایگریشن را (بی‌خطر و idempotent، یعنی جدول‌های موجود را دوباره
   نمی‌سازد) اجرا کند و بعد سرور را راه بیندازد. کافی‌ست لاگ‌های اولین دیپلوی را
   ببینید تا خط «مایگریشن با موفقیت انجام شد» را ببینید.

   > اگر بعداً یک سرور با پلن پولی (که Shell دارد) استفاده کردید، همچنان می‌توانید
   > مایگریشن را دستی هم با `npm run migrate` از Shell اجرا کنید — تفاوتی ندارد.

### مرحله ۳ — بررسی هر ۱۲ دقیقه (cron-job.org)

چون سرویس رایگان Render می‌خوابد، به‌جای زمان‌بند داخلی از یک پینگ‌کننده‌ی خارجیِ
رایگان استفاده می‌کنیم که هم سرور را بیدار نگه می‌دارد هم بررسی را صدا می‌زند:

1. در [cron-job.org](https://cron-job.org) ثبت‌نام کنید (رایگان، بدون کارت).
2. یک Cron Job جدید بسازید:
   - **URL**: `https://<آدرس-سرویس-شما>.onrender.com/api/cron/run-check`
   - **Method**: `POST`
   - **Header**: `X-Cron-Secret: <همان مقدار CRON_SECRET>`
   - **Schedule**: هر ۱۲ دقیقه (`*/12 * * * *`)

> از آنجا که این فاصله (۱۲ دقیقه) کوتاه‌تر از آستانه‌ی خواب Render (۱۵ دقیقه)
> است، همین یک Cron Job سرور را همیشه بیدار هم نگه می‌دارد — نیازی به یک پینگ‌کننده‌ی
> جداگانه برای `/health` نیست.

> ⚠️ با این فاصله‌ی کوتاه، تعداد اجرای Puppeteer در روز حدود ۱۲۰ برابر حالت
> ساعتی می‌شود (هر ۱۲ دقیقه = ۱۲۰ بار در روز به‌جای ۲۴ بار). اگر تعداد لینک‌های
> ثبت‌شده‌تان زیاد شود، ممکن است به سقف پهنای‌باند/CPU رایگان Render یا Supabase
> نزدیک‌تر شوید، و برخی سایت‌ها ممکن است رفتار ربات مکرر را نامعمول تشخیص دهند.
> اگر لازم شد، مقدار `CHECK_CRON` را از طریق environment variable در Render
> (بدون نیاز به تغییر کد) به هر فاصله‌ی دیگری برگردانید.

### مرحله ۴ — فرانت‌اند

ساده‌ترین راه: در همان پروژه‌ی Render → **New → Static Site** → همان ریپو را
انتخاب کنید، **Publish Directory** را روی پوشه‌ی `frontend` بگذارید. بعد از دیپلوی،
قبل از تگ `<script>` اصلی در `index.html` این خط را اضافه/تنظیم کنید تا به آدرس
واقعی بک‌اندتان وصل شود:

```html
<script>window.RASADLINK_API_BASE = 'https://<آدرس-سرویس-شما>.onrender.com';</script>
```

سپس مقدار `CORS_ORIGIN` در متغیرهای محیطی بک‌اند را به آدرس همین فرانت‌اند
(مثلاً `https://your-frontend.onrender.com`) تغییر دهید تا مرورگر درخواست‌ها را
بلاک نکند.

### جمع‌بندی مسیر رایگان

بعد از این ۴ مرحله: فرانت روی یک آدرس رایگان بالاست، بک‌اند روی Render رایگان
اجراست، دیتابیس و عکس‌ها روی Supabase رایگان ذخیره می‌شوند، و هر ۱۲ دقیقه
cron-job.org رایگان بک‌اند را برای بررسی لینک‌ها صدا می‌زند — بدون پرداخت هیچ هزینه‌ای.

اگر بعداً خواستید از این محدودیت‌ها (خواب سرویس، ۵۰۰ مگابایت دیتابیس و ...) عبور
کنید، فقط کافی‌ست پلن Render/Supabase را ارتقا دهید؛ هیچ‌کدام از تنظیمات یا کد
عوض نمی‌شود.



| متد | مسیر | توضیح |
|---|---|---|
| GET | `/api/links` | لیست لینک‌ها (پارامتر اختیاری `?category=`) |
| POST | `/api/links` | افزودن لینک `{ url, title, cat }` |
| GET | `/api/links/:id` | جزئیات یک لینک |
| PATCH | `/api/links/:id` | ویرایش `{ title?, category?, status? }` (`status`: `active`/`paused`) |
| DELETE | `/api/links/:id` | حذف لینک (و اسکرین‌شات مرتبط) |
| POST | `/api/links/:id/check` | بررسیِ فوریِ همین لینک (خارج از چرخه‌ی خودکار) |
| GET | `/api/links/:id/history` | تاریخچه‌ی تغییرات محتواییِ یک لینک |
| POST | `/api/links/:id/dismiss` | خواندن‌شده کردن اعلان‌های یک لینک (خاموش کردن بج) |
| GET | `/api/categories` | دسته‌بندی‌ها به همراه تعداد لینک هر کدام |
| GET | `/api/notifications` | لیست اعلان‌ها |
| PATCH | `/api/notifications/:id/read` | خواندن‌شده کردن یک اعلان |
| PATCH | `/api/notifications/read-all` | خواندن‌شده کردن همه‌ی اعلان‌ها |
| GET | `/api/history` | تاریخچه‌ی تغییرات همه‌ی لینک‌ها (برای منوی کناری) |
| POST | `/api/cron/run-check` | اجرای فوری چرخه‌ی بررسی — برای اتصال یک cron خارجی رایگان (نیازمند هدر `X-Cron-Secret`) |
| GET | `/uploads/screenshots/...` | فایل‌های اسکرین‌شات وقتی `STORAGE_DRIVER=local` است (سرو استاتیک) |
| GET | `/health` | سلامت سرویس |

## گام‌های بعدیِ پیشنهادی (اختیاری)

- افزودن احراز هویت اگر قرار شد چندکاربره شود.
- گرفتن اسکرین‌شات فقط از ناحیه‌ای که تغییر کرده (نه کل صفحه) با هایلایت‌کردن آن.
- استفاده از یک صف (مثل BullMQ) به‌جای اجرای مستقیم در پروسه‌ی cron، برای تعداد
  زیاد لینک یا مقیاس افقی روی چند سرور.
